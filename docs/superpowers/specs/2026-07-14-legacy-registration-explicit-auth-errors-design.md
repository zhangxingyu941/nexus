# 遗留账号注册与明确认证错误设计

## 背景

旧版本可以创建没有密码、邮箱也未验证的用户。此类用户再次走邮箱注册时，现有逻辑会把它当成重复账号并返回笼统的“无法创建账号”。认证接口还合并了账号不存在、密码错误、邮箱未验证、验证码错误和验证码过期等原因，用户无法判断下一步操作。

## 目标

- 允许未验证且没有密码的遗留账号完成邮箱注册，并保留原用户 ID、工作区和文档。
- 所有可预期的认证业务错误都返回明确原因和可执行的下一步提示。
- 前端直接展示服务端错误，不把数据库、SMTP、堆栈或配置值暴露给用户。
- 保持验证码 10 分钟有效和同账号 60 秒重发冷却。

## 明确取舍

产品要求错误原因完全明确，因此接口会区分邮箱未注册、邮箱已注册、密码错误、邮箱未验证等账号状态。这样会允许请求方判断邮箱是否存在，降低账号防枚举能力。该取舍由产品要求明确接受，限流和认证审计继续保留。

## 服务端设计

### 遗留账号升级

注册发现同邮箱用户时按状态处理：

- 未验证且没有密码：在事务和账号行锁内写入新密码哈希、注册姓名和邮箱验证码，保留原用户 ID、成员关系、工作区和文档。
- 未验证且已有密码：密码一致时按冷却规则重发；密码不一致时提示使用首次注册密码或找回密码。
- 已验证且已有密码：提示邮箱已注册，请登录或找回密码。
- 已验证但没有密码：提示账号由其他登录方式创建，请使用找回密码设置密码或继续使用第三方登录。

### 错误契约

认证接口继续返回 JSON `{ error: string, retryAfterSeconds?: number }`。预期业务错误使用稳定中文提示和对应状态码：

- `400`：字段缺失、格式错误、密码长度错误、验证码格式错误或验证码错误。
- `401`：密码错误。
- `403`：邮箱尚未验证。
- `404`：邮箱未注册或没有可用验证码。
- `409`：邮箱已注册或未完成注册的密码不一致。
- `410`：验证码已过期。
- `429`：限流或验证码冷却，并返回明确剩余秒数。
- `503`：数据库、限流服务或邮件服务不可用；只说明哪个服务失败，不返回内部异常文本。

验证码验证会先校验输入格式，再定位账号和最新未消费验证码，并分别判断未请求、已过期和内容错误。登录会分别判断邮箱格式、账号不存在、未设置密码、邮箱未验证和密码错误。

验证码事务统一先锁用户行、再锁验证码行，避免验证与重发并发时产生反向锁序。认证审计采用尽力写入，审计存储故障不会覆盖已经确定的业务响应。SMTP 配置初始化异常由路由转换为明确 `503`，不会落入框架默认错误页。

找回密码不再隐藏账号不存在。邮件发送改为等待 SMTP 结果后再响应，以便发送失败时返回明确 `503`；这会增加响应时间差异，但符合本次完整错误提示要求。

## 前端设计

- 认证表单继续使用现有错误提示区域展示 API 的明确中文原因。
- 请求无法到达服务端时提示检查网络；响应不是合法 JSON 时提示认证服务响应异常。
- 找回密码成功后明确显示验证码已发送到规范化邮箱，不再使用“如果账号存在”。
- 收到包含 `retryAfterSeconds` 的 `429` 时继续启动或校正倒计时。

## 测试

- 存储层覆盖遗留账号升级且用户 ID、工作区不变。
- 覆盖已注册、未完成注册密码不一致、邮箱不存在、邮箱未验证、密码错误以及验证码未请求、错误、过期。
- 路由层覆盖各状态码、错误文本、冷却秒数和 SMTP 失败。
- UI 覆盖明确错误展示、网络错误、异常响应和找回密码成功文案。
