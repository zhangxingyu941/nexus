# 生产加固设计

## 1. 目标

将编辑器从当前的开发身份和单进程协作模型迁移到面向生产的基线，同时不替换现有的用户 ID、工作区所有权、文档数据或角色分配。

工作分四个有序增量交付：

1. 安全的混合认证。
2. 持久的多实例协作。
3. 可复现的容器部署。
4. 端到端验收和运维检查。

## 2. 约束

- 现有的 `app_users`、工作区成员关系、会话和文档所有权必须保持有效。
- PostgreSQL 仍然是用户、会话、权限、工作区数据和协作快照的权威来源。
- Redis 是生产限流和协作发布/订阅所必需的，但当 Redis 不可用时，会话读取继续回退到 PostgreSQL。
- GitHub OAuth 在运行时是可选的。除非同时配置了 GitHub 凭据，否则登录操作将被隐藏。
- SMTP 在开发环境中是可选的，在生产环境中是验证和密码重置投递所必需的。
- 查看器绝不能加入可写的 Yjs 房间。
- 密钥属于 `.env` 或部署密钥，永远不在提交的示例配置中。
- Node.js 18.20.2 仍然是受支持的本地运行时。

## 3. 认证架构

### 3.1 库

- 使用 `@node-rs/argon2` 进行 Argon2id 密码哈希和验证。
- 使用 `arctic` 进行 GitHub OAuth 授权、状态验证和 PKCE 处理。
- 使用 `nodemailer` 进行通用 SMTP 投递。
- 使用现有的 PostgreSQL 和 Redis 客户端，而不是添加第二个持久化抽象。

OAuth 协议细节和密码学不能从头实现。

### 3.2 数据库变更

扩展 `app_users`，添加可为空的认证字段，使现有开发用户迁移时不丢失数据：

- `password_hash TEXT`
- `email_verified_at BIGINT`
- `updated_at BIGINT`

添加 `oauth_accounts`：

- 提供者和提供者账户 ID 构成主要身份。
- 每行引用一个现有的 `app_users` 用户。
- 提供者邮箱保留用于审计和账户关联检查。

添加 `auth_tokens`：

- 仅存储 SHA-256 令牌哈希。
- 支持 `verify-email` 和 `reset-password` 用途。
- 强制过期和一次性消费。

现有的 `auth_sessions` 仍然是浏览器和 WebSocket 会话的权威来源。密码和 OAuth 登录最终都创建与应用程序当前使用的相同的不透明、哈希的会话令牌。

### 3.3 邮箱和密码流程

注册需要显示名称、规范化邮箱和至少 12 个字符的密码。密码在插入前使用 Argon2id 哈希。注册创建工作区并发送验证链接，但不在 API 响应中暴露原始令牌。

登录需要邮箱和密码。错误使用单一通用消息，使 API 不会泄露邮箱是否存在。成功的登录轮换到新的不透明会话令牌。重复失败按规范化邮箱和客户端 IP 进行限流。

邮箱验证和密码重置使用短期、一次性使用的哈希令牌。密码重置在颁发替换登录会话之前撤销用户的所有现有会话。

在开发中，如果 SMTP 缺失，投递会将验证 URL 写入服务器日志。在生产中，缺少 SMTP 配置是需要邮件的端点的启动/配置错误。

### 3.4 GitHub OAuth 流程

浏览器从 `/api/auth/oauth/github` 开始。服务器创建 state、PKCE 验证器和短期 HttpOnly SameSite=Lax 事务 Cookie，然后重定向到 GitHub。

回调在交换代码之前验证 state 和 PKCE。它请求 GitHub 用户和已验证邮箱端点。账户关联遵循以下规则：

1. 现有的提供者/账户 ID 登录其关联的用户。
2. 已验证的 GitHub 邮箱可以关联到具有相同规范化邮箱的现有本地用户。
3. 未验证或缺失的 GitHub 邮箱被拒绝。
4. 否则创建新的已验证用户和个人工作区。

回调创建与密码登录相同的数据库支持的应用程序会话，然后重定向到 `/`。

### 3.5 遗留用户迁移

具有 `password_hash IS NULL` 的现有用户保留所有数据和成员关系。他们不能使用密码登录，直到完成密码重置，但可以通过匹配的已验证 GitHub 邮箱登录。UI 在不向未认证调用者标识账户存在的情况下解释这一点。

### 3.6 限流和审计

Redis 存储注册、登录、密码重置和 OAuth 回调失败的有界计数器。键包含邮箱或 IP 组件的单向哈希。如果限流存储不可用，生产会拒绝认证尝试；开发可以使用进程本地回退。

认证事件记录在 `auth_audit_events` 表中，包含事件类型、已知时的用户 ID、哈希 IP、时间戳和成功标志。密码、原始令牌、会话令牌、OAuth 代码和完整 IP 地址永远不会被记录。

## 4. 认证 UI

现有的灰色/白色 shadcn 视觉语言保持不变。匿名界面变为紧凑的认证表面，包含：

- 登录和注册选项卡。
- 带有无障碍标签的邮箱和密码字段。
- 忘记密码流程。
- 验证/重置状态屏幕。
- 仅当服务器报告 GitHub OAuth 已配置时才显示的 GitHub 按钮。
- 通用、不泄露信息的错误消息。

已认证的编辑器行为和工作区布局不变。

## 5. 持久协作

### 5.1 PostgreSQL 持久化

添加 Yjs 房间快照和增量更新表。房间键保持为 `workspace:<workspaceId>:document:<documentId>` 范围。

当房间首次打开时，协作服务器加载最新的压缩快照和后续更新，然后将它们应用于 Yjs 文档。每个本地更新作为 `BYTEA` 追加。在达到有界的数量或字节大小的更新后，服务器在事务中使用 Yjs 更新 API 将它们压缩为一个快照。

持久化适配器拥有加载、追加、压缩和删除操作。WebSocket 授权层保持独立。

### 5.2 Redis 发布/订阅

每个协作实例使用一个 Redis 订阅者连接和一个发布者连接。更新发布到带有实例 ID 和更新载荷的房间特定通道。实例忽略自己的消息，应用带有标记来源的远程更新，并且不重新发布这些远程更新。

感知消息使用单独的临时通道，永远不会存储在 PostgreSQL 中。Redis 中断使当前实例保持可用但标记跨实例同步降级；PostgreSQL 更新持久化继续。

### 5.3 关闭和恢复

优雅关闭停止接受升级、刷新待处理的更新写入、断开 Redis、关闭房间，然后关闭 PostgreSQL。在不干净的重启中，持久化的更新在接受客户端之前重建房间。

## 6. 部署

添加包含以下内容的 Docker Compose 堆栈：

- 带有命名/绑定卷的 PostgreSQL。
- 带有追加写入持久化和健康检查的 Redis。
- Next.js 应用程序。
- 协作服务。

应用程序和协作容器共享环境配置但作为独立进程运行。健康检查覆盖 PostgreSQL 连接、Redis 连接、Next.js API 健康和协作 HTTP 健康。服务仅在所需依赖健康后启动。

提供面向生产的 `.env.example`（不含真实密钥）和本地覆盖示例。数据库迁移作为显式的一次性命令运行，而不是在每个 Web 进程中隐式运行。

## 7. 测试

实施遵循测试优先的增量：

- 密码哈希边界、令牌生命周期、账户关联、限流和配置验证的单元测试。
- 架构迁移、遗留用户、会话撤销、OAuth 账户唯一性、Yjs 更新恢复和压缩的 PostgreSQL 集成测试。
- 注册、登录、验证、重置、OAuth 启动/回调失败和 Cookie 行为的 API 测试。
- 持久化恢复、双实例 Redis 传播、无回显循环、租户隔离和查看器拒绝的协作测试。
- 注册/登录、使用开发邮件捕获的密码重置、编辑器加载和双浏览器协作的 Playwright 测试。
- 最终关卡：完整 Vitest 套件、TypeScript、`git diff --check`、生产构建、Compose 健康、数据库冒烟和浏览器桌面/移动截图。

## 8. 交付顺序

1. 数据库迁移和认证域服务。
2. 邮箱/密码 API 和 UI。
3. 可选的 GitHub OAuth 和账户关联。
4. Redis 限流和认证审计。
5. PostgreSQL Yjs 持久化。
6. Redis 协作发布/订阅。
7. Docker Compose 和运维文档。
8. 端到端和视觉验收。

每个增量必须使现有编辑器保持可用，并在下一个增量开始之前通过聚焦测试。

## 9. 完成标准

生产加固阶段在以下条件下完成：

- 邮箱/密码注册、验证、登录、登出、重置和会话撤销正常工作。
- GitHub OAuth 在配置时正常工作，在未配置时从 UI 中隐藏。
- 现有用户和工作区权限在迁移后存活。
- 协作内容在服务器重启后存活。
- 两个协作服务器实例通过 Redis 交换更新，无循环或租户泄漏。
- Compose 堆栈从文档化的配置启动且所有健康检查通过。
- 自动化和浏览器验收套件通过，无已知的严重或重要发现。
