# 多文档编辑器布局设计

## 概览

本次迭代把当前单文档块编辑器升级为多文档工作区。页面采用左侧文档列表、右侧专属编辑区域的布局，让用户可以创建多个文档，并在选中文档后编辑该文档自己的富文本块内容。

本次仍保持本地单机版本，不引入登录、后端、协同、文档删除、搜索、拖拽排序或复杂权限。

## 产品范围

用户进入应用后看到一个两栏工作区：

- 左侧是文档管理区，展示文档列表和新建文档入口。
- 右侧是当前文档的专门编辑区，展示文档标题、保存状态和块编辑画布。
- 新建文档后，应用自动选中新文档，并在右侧显示空白编辑区域。
- 点击左侧文档后，右侧编辑区切换到对应文档。
- 编辑右侧内容只影响当前选中的文档。
- 刷新页面后恢复文档列表、文档内容和当前选中文档。

## 布局设计

### 左侧文档区

左侧固定为工作区导航栏，宽度保持紧凑，主要用于管理文档：

- 顶部显示应用名称或工作区名称。
- 明显位置提供 `新建文档` 按钮。
- 文档列表按更新时间倒序展示。
- 当前文档使用背景色、左侧强调线或加粗标题区分。
- 每条文档显示标题；如果标题为空，显示 `未命名文档`。
- 可选展示最近更新时间，但不在第一版加入复杂摘要。

### 右侧编辑区

右侧是文档的主要工作区域：

- 顶部显示当前文档标题和保存状态。
- 下方是专门的编辑画布，承载现有块编辑能力。
- 编辑画布保持宽度约束，避免长行影响阅读。
- 空文档始终保留一个可输入段落块。
- 块操作控件继续支持类型切换、新增、删除、上下移动和待办勾选。

## 数据模型

当前模型只有一个 `EditorDocument`。本次新增工作区模型：

```ts
export interface EditorWorkspace {
  documents: EditorDocument[];
  activeDocumentId: string;
  updatedAt: number;
}
```

每个 `EditorDocument` 继续沿用现有结构：

```ts
export interface EditorDocument {
  id: string;
  title: string;
  blocks: Block[];
  updatedAt: number;
}
```

新增文档时创建一个包含空段落块的文档，并把 `activeDocumentId` 设置为该文档 ID。

## 状态操作

新增一组纯函数管理工作区状态：

- 创建默认工作区。
- 新建文档并自动选中。
- 切换当前文档。
- 更新当前文档的块内容。
- 修改当前文档的块类型。
- 切换当前文档的待办状态。
- 在当前文档中新增、删除、移动块。

现有单文档操作函数继续保留，并被工作区操作复用。这样可以减少重写风险，也方便后续扩展到服务端文档列表。

## 持久化

IndexedDB 从保存单个默认文档升级为保存整个工作区：

- `loadWorkspace(): Promise<EditorWorkspace | null>`
- `saveWorkspace(workspace: EditorWorkspace): Promise<void>`
- `clearWorkspace(): Promise<void>`

为了兼容已存在的本地数据，加载时可以做一次轻量迁移：

- 如果找到新的工作区数据，直接使用。
- 如果只找到旧的单文档数据，把它包装成一个包含单文档的工作区。
- 如果没有任何数据，创建默认工作区和默认文档。

自动保存仍然使用防抖逻辑，保存状态继续显示为：

- `已保存`
- `保存中`
- `未保存`
- `保存失败`

## 组件调整

建议拆分为以下组件：

- `EditorPage`：加载工作区、保存工作区、管理当前文档。
- `WorkspaceSidebar`：渲染左侧文档列表和新建文档按钮。
- `DocumentEditor`：渲染右侧标题、保存状态和块列表。
- `EditorToolbar`：可保留为右侧顶部状态栏，或并入 `DocumentEditor`。
- `BlockList`、`BlockRow`、`RichTextBlockEditor`、`TodoBlockEditor`：继续复用现有块编辑组件。

## 交互细节

- 点击 `新建文档` 后，左侧列表立即出现新文档，右侧切换到该文档。
- 点击文档列表项时，只切换当前文档，不触发额外弹窗。
- 文档标题第一版先由系统生成 `未命名文档`；标题编辑可以放到后续迭代。
- 如果用户删除当前文档内最后一个块，仍自动补一个空段落块。
- 小屏幕下左侧文档区堆叠到上方，右侧编辑区在下方，保证移动端可用。

## 视觉方向

整体保持工具型产品界面：

- 左侧导航栏使用浅底、细边框和紧凑间距。
- 右侧编辑区域更宽、更安静，像专门的写作画布。
- 操作按钮继续使用图标和中文可访问标签。
- 聚焦、悬停、选中状态要清晰，但避免过度装饰。
- 不使用营销式大标题或落地页布局。

## 错误处理

- 工作区加载失败时创建默认工作区，并显示保存失败状态。
- 工作区保存失败时保留内存中的编辑内容，用户仍可继续编辑。
- 当前文档 ID 无效时，回退选中第一个文档；如果没有文档，则创建默认文档。
- 文档列表为空时，自动创建一个默认文档，避免编辑区空白。

## 测试策略

模型测试覆盖：

- 默认工作区包含一个默认文档。
- 新建文档会追加文档并选中新文档。
- 切换文档只改变 `activeDocumentId`。
- 当前文档的块操作不会影响其他文档。
- 当前文档 ID 无效时能回退到有效文档。

持久化测试覆盖：

- 保存和加载工作区。
- 清空工作区。
- 从旧单文档数据迁移到工作区数据。

组件测试覆盖：

- 页面渲染左侧文档列表和右侧编辑区。
- 点击新建文档后文档数量增加，并切换到新文档。
- 点击不同文档后，右侧显示对应文档内容。
- 编辑当前文档内容时，其他文档内容不变。

## 验收标准

- 应用启动后显示左侧文档列表和右侧编辑区域。
- 用户可以通过 `新建文档` 创建多个文档。
- 用户可以在左侧切换文档。
- 右侧编辑区始终对应当前选中的文档。
- 文档内容可以继续使用段落、标题、待办块编辑。
- 刷新浏览器后恢复文档列表、当前文档和编辑内容。
- `npm test -- --run` 通过。
- `npm run build` 通过。
