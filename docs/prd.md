## 2. 项目背景

本项目最初是面向学习和作品展示的现代协作编辑器练手项目，现已完成从本地编辑器到带认证、权限、持久化和实时协同的完整技术验证。后续产品定位调整为服务 3–30 人产品/项目团队的协作文档工作台，同时保留对块编辑、CRDT 和全栈架构的研究价值。

项目涉及两个主要技术难点：

- 灵活的数据模型：一切内容抽象为 Block，支持段落、标题、待办、图片等不同类型，并为后续嵌套块、拖拽排序、页面树扩展留下空间。
- 分布式数据一致性：多人同时编辑同一页面时，通过 CRDT 技术同步内容，减少手写冲突解决逻辑。

## 3. 产品目标

### 3.1 长期目标

构建一个支持块编辑、实时协同、持久化存储和基础权限控制的协作编辑应用。

### 3.2 阶段目标

第一阶段优先交付一个可本地运行、可实际编辑的单机版块编辑器，验证编辑体验和块数据模型；之后再逐步引入协同后端、数据库持久化和进阶功能。

### 3.3 成功标准

- 用户可以创建和编辑由多个块组成的页面。
- 用户可以新增、删除、修改、排序基础块。
- 页面内容可以本地保存并在刷新后恢复。
- 后续阶段中，两个浏览器窗口可以实时同步同一页面内容。
- 数据模型清晰，能够扩展到嵌套块、协同编辑和数据库持久化。

## 4. 用户与使用场景

### 4.1 目标用户

- 主要用户：3–30 人产品、研发、设计或项目团队，需要共同沉淀文档、跟踪任务并围绕内容协作。
- 工作区所有者：负责创建团队空间、邀请成员、设置权限和管理共享范围。
- 编辑成员：负责创建文档、编辑内容、处理任务、评论和 @提及。
- 查看成员或访客：以只读方式查看被授权内容并参与受限协作。
- 次要用户：希望学习富文本编辑器、CRDT、实时协同和树形数据建模的开发者。

### 4.2 核心场景

- 用户打开编辑器，创建或继续编辑一个文档页面。
- 用户通过块结构组织内容，例如标题、段落和待办事项。
- 用户调整块顺序，快速组织页面结构。
- 在协同阶段，多个用户打开同一页面并实时看到彼此编辑结果。

## 5. 产品范围

### 5.1 第一阶段 MVP 范围

第一阶段聚焦单机版块编辑器，不接入后端协同服务。

必须包含：

- 基础编辑页面。
- 三种基础块：段落、标题、待办。
- 块的新增、删除、编辑。
- 块的排序，优先支持简单上移/下移；如果实现成本可控，再支持拖拽排序。
- 本地持久化，优先使用 IndexedDB。
- 页面刷新后恢复编辑内容。

暂不包含：

- 多人实时协同。
- 登录、权限和用户体系。
- PostgreSQL、Redis、对象存储。
- 历史版本。
- 图片、文件、表格、看板等复杂块。

> 说明：以上是第一阶段 MVP 的历史排除范围。当前项目已推进到 M5，这些能力均已有实现；详见第 14 节当前实现状态。

### 5.2 第二阶段范围：协同雏形

- 引入 Yjs 作为 CRDT 引擎。
- 引入 WebSocket 协同服务。
- 两个浏览器窗口打开同一页面时可以同步文本内容。
- 实现基础在线状态或光标展示。
- 暂不处理复杂权限和长期历史版本。

### 5.3 第三阶段范围：后端与持久化

- 引入 Node.js 后端服务。
- 将页面和块数据保存到 PostgreSQL。
- 设计块的树形关系存储结构。
- 支持页面初次加载时从数据库恢复数据。
- 增加简单用户和页面归属关系。

### 5.4 第四阶段范围：进阶能力

- 嵌套块。
- 图片或文件块。
- Slash Command。
- 历史版本。
- 离线编辑与重连同步。
- 大页面性能优化，例如虚拟滚动。

## 6. 功能需求

### 6.1 页面编辑

- 用户进入应用后默认看到一个编辑页面。
- 页面由多个块组成。
- 页面内容为空时显示可编辑的起始块。
- 用户输入内容后，块内容实时更新到应用状态。

### 6.2 块类型

第一阶段支持三种块类型：

- 段落块：用于普通文本。
- 标题块：用于一级或二级标题，第一阶段可先支持单一标题样式。
- 待办块：包含勾选状态和文本内容。

后续可扩展：

- 图片块。
- 文件块。
- 表格块。
- 看板块。
- 嵌套容器块。

### 6.3 块操作

第一阶段支持：

- 在当前块后新增块。
- 删除当前块。
- 修改块类型。
- 编辑块内容。
- 调整块顺序。

建议交互：

- 回车创建新段落块。
- 空块退格删除当前块或合并到前一个块。
- 待办块支持勾选和取消勾选。

### 6.4 本地持久化

- 编辑内容需要保存到浏览器本地。
- 刷新页面后恢复上次编辑内容。
- 本地存储结构需要与后续服务端数据模型尽量保持一致。

### 6.5 实时协同

第二阶段支持：

- 多个客户端加入同一页面房间。
- 一个客户端编辑文本后，其他客户端实时收到更新。
- 使用 Yjs 自动处理并发编辑冲突。
- 通过 awareness 机制同步光标或用户在线状态。

### 6.6 后端 API

第三阶段可提供 REST API：

- `POST /api/pages`：创建页面。
- `GET /api/pages/:pageId/blocks`：获取页面下所有块。
- `POST /api/blocks/:blockId/move`：移动块到新位置。
- `GET /api/history/:pageId`：获取页面历史版本。

第三阶段可提供 WebSocket API：

- 连接地址：`ws://host/collab/page_:pageId`
- 消息类型：`yjs-update`
- 消息类型：`awareness`

## 7. 数据模型

### 7.1 Block 基础结构

```json
{
  "id": "block_uuid_1",
  "type": "paragraph",
  "parent": "page_uuid_123",
  "children": ["block_uuid_2", "block_uuid_3"],
  "content": {
    "text": "这是一个段落",
    "richText": [
      { "text": "这是一个", "bold": true },
      { "text": "段落" }
    ]
  },
  "createdBy": "user_1",
  "updatedAt": 1678886400000,
  "version": 15
}
```

### 7.2 第一阶段本地模型

第一阶段可以简化为：

```ts
type BlockType = "paragraph" | "heading" | "todo";

interface Block {
  id: string;
  type: BlockType;
  content: string;
  checked?: boolean;
  parentId: string | null;
  children: string[];
  createdAt: number;
  updatedAt: number;
}
```

### 7.3 后续数据库模型

后续服务端建议采用闭包表加物化路径组合：

- `blocks`：存储块的基础属性和内容。
- `block_relationships`：存储祖先、后代和层级深度。
- `pages`：存储页面元数据。
- `users`：存储用户基础信息。

## 8. 技术方案

### 8.1 推荐技术栈

第一阶段：

- 前端框架：React
- 编辑器内核：TipTap
- 本地存储：IndexedDB
- 构建工具：Vite
- 测试：Vitest + Testing Library

第二阶段：

- 协同引擎：Yjs
- 实时通信：WebSocket 或 y-websocket
- 协同服务：Node.js

第三阶段：

- 后端：Node.js
- 数据库：PostgreSQL
- 可选缓存：Redis
- 可选协同框架：Hocuspocus

### 8.2 技术选型理由

- TipTap 基于 ProseMirror，生态成熟，适合快速实现富文本编辑能力。
- Yjs 是成熟 CRDT 库，可以降低多人协同冲突处理复杂度。
- IndexedDB 适合第一阶段本地持久化，避免过早引入后端复杂度。
- PostgreSQL 适合后续存储页面、块和树形关系。

## 9. 非功能需求

### 9.1 性能

- 第一阶段应支持至少 200 个块的流畅编辑。
- 后续阶段需要关注大量块渲染性能，必要时引入 React.memo 或虚拟滚动。

### 9.2 可扩展性

- 块类型需要通过清晰的数据结构扩展。
- 编辑器状态和持久化逻辑需要隔离，便于从 IndexedDB 迁移到服务端。
- 协同逻辑不应污染基础块编辑逻辑。

### 9.3 可靠性

- 本地保存失败时需要保留内存中的编辑内容。
- 协同阶段网络断开时，应提示离线状态，并在恢复后重新同步。

### 9.4 可测试性

- 块 reducer 或状态操作函数需要覆盖单元测试。
- 本地存储适配层需要覆盖保存和恢复测试。
- 编辑器关键交互需要覆盖组件测试。

## 12. 风险与约束

- 富文本编辑器交互复杂，不建议从零实现 contenteditable。
- CRDT 算法复杂，不建议自研协同算法。
- 第一阶段若过早引入后端、权限和数据库，会拖慢 MVP 验证。
- Yjs 二进制文档与结构化数据库之间存在同步复杂度，第三阶段需要专门设计。
- 嵌套块和拖拽排序会显著增加状态管理复杂度，应在基础编辑稳定后再做。

## 15. 后续产品规划（2026-07-14）

### 15.1 产品定位

面向 3–30 人产品/项目团队，提供一个把文档、轻量任务和围绕内容的协作放在同一工作区的工具。产品首先解决“团队成员能否顺利进入同一空间、找到正确内容、按权限协作并可靠保存”的问题，再增强编辑深度和 AI。

后续规划包含三条能力线：

- `P0 协作闭环`：M6 团队、工作区、邀请和成员生命周期已实现；下一优先项是真实分享和页面权限，且 M6.2 仍需完成环境级最终回归。
- `P1 编辑与知识发现`：结构化富文本、拖拽、多选、导入导出、通知、@提及和服务端搜索。纳入规划，P0 稳定后推进。
- `P2 AI 工作台`：写作辅助、总结、任务提取、语义检索和权限感知问答。暂不优先，必须建立在稳定权限和检索之上。

### 15.2 产品原则

- 团队闭环优先于功能数量：邀请、授权、访问、协作和退出流程必须完整。
- 权限由服务端裁决：前端隐藏按钮不能替代 API、文件和 WebSocket 鉴权。
- 内容只有一个权威模型：富文本、结构化块、PostgreSQL 和 Yjs 之间必须定义清晰边界。
- 用户操作可恢复：删除、成员变更、权限调整和历史恢复需要明确确认及审计。
- AI 不绕过权限：检索、上下文拼接、缓存和引用都必须按当前用户权限过滤。
- 复杂数据库能力、企业治理和自动化 Agent 不进入当前优先阶段。

### 15.3 核心用户旅程

1. 用户注册或登录，创建或加入一个团队工作区。
2. 所有者通过邮箱邀请成员，并可查看待接受、已加入和已撤销状态。
3. 成员切换工作区，通过页面树、搜索、最近访问或通知找到内容。
4. 多人按 owner/editor/viewer 权限编辑、评论、@提及和处理任务。
5. 文档可通过团队权限、指定成员或只读分享链接访问。
6. 用户从通知中心回到评论、提及或分配给自己的任务。
7. 后续可在当前权限范围内使用 AI 总结、改写或查询工作区知识。

## 16. M6：团队与工作区闭环（P0）

### 16.1 目标

把现有“个人工作区 + 已注册用户直接授权”升级为可供真实小团队使用的成员和工作区体系。

### 16.2 功能范围

#### 第一批已完成：多工作区基础

- 用户可以创建、查看、切换和重命名多个工作区。
- 工作区切换必须由用户主动触发，收到邀请或被添加成员时不能自动替换当前工作区。
- 侧栏显示当前工作区名称、用户角色和切换入口。
- 所有 API、IndexedDB 缓存键、Yjs room、文件对象键和历史版本都带工作区作用域。
- IndexedDB v2 自动迁移旧本地数据，默认目录名称为 `Nexus 工作区`。
- PostgreSQL 使用 `workspace_document_preferences` 记录每名成员在每个工作区的活动文档。

#### M6.2 已实现：邀请、成员与删除生命周期

**邮件邀请（M6.2A）**

- owner 可邀请尚未注册或已经注册的邮箱，并选择 editor 或 viewer；同一邮箱的有效邀请不能重复创建。
- 邀请包含随机高熵原始令牌，但数据库、审计、API 响应和浏览器存储只保留其 HMAC 或签名上下文。邀请 24 小时过期，邮件链接先解析为 30 分钟 HttpOnly 邀请上下文。
- 支持查看、重发（轮换令牌）、撤销和投递状态。SMTP 投递失败不回滚已创建的邀请，而是返回明确告警，方便重发。
- 未登录收件人完成注册或登录后回到验证过的邀请路径；接受后才显式切换到目标工作区。

**成员生命周期（M6.2B）**

- owner 可查看成员、调整角色、移除成员并转让所有权；工作区可以有多个 owner。
- 普通成员可主动退出。最后一名 owner 不能被降级、移除或退出，所有权转让在事务内完成。
- 角色降低、移除、退出、所有权转让和工作区删除会广播访问失效。服务端在下一次 REST/文件请求拒绝权限，并关闭该用户受影响工作区的既有协作 WebSocket；其他工作区会话保持有效。

**删除、回收站与永久清理（M6.2C）**

- owner 必须输入精确工作区名称才能软删除。删除会返回删除摘要、撤销待处理邀请并保留成员关系，便于恢复。
- 仅仍保留 owner 关系的用户可在回收站查看和恢复 tombstone；保留期固定为删除后 7 天。
- 任意生命周期请求会低优先级处理少量到期 tombstone。清理严格先删除 `workspaceId/` 对象存储前缀，再删除数据库行；对象删除或数据库删除失败时记录脱敏错误并保留 tombstone，供后续请求重试。

#### 下一批：账号设置

- 显示名称和密码变更、其他登录会话查看/撤销，以及经过独立验证的主邮箱变更仍未实现。

### 16.3 数据与 API

第一批已交付 API：

- `GET|POST /api/workspaces`
- `GET|PUT|PATCH /api/workspaces/:workspaceId`
- `POST /api/workspaces/:workspaceId/select`
- `GET|POST /api/workspaces/:workspaceId/members`
- `GET|POST /api/workspaces/:workspaceId/history/:documentId`
- `POST /api/files`，表单包含 `workspaceId`

第一批已交付数据结构包括 `workspace_document_preferences`，并将工作区成员、内容、版本、文件和规范 Yjs room `workspace:{workspaceId}:document:{documentId}` 绑定到同一作用域。

M6.2 已交付数据结构：

- `workspace_invites`：邀请 ID、工作区、邮箱、角色、令牌 HMAC、状态、过期时间、邀请人、收件人和投递状态。
- `workspace_audit_events`：邀请、成员和工作区删除/恢复的脱敏审计事件。
- `editor_workspaces.deleted_at` 与 `purge_after`：受约束的 7 天 tombstone；永久清理成功前保留记录。

M6.2 已交付或调整的 API：

- `GET|POST /api/workspaces/:workspaceId/invites`、`POST /api/workspaces/:workspaceId/invites/:inviteId/resend`、`DELETE /api/workspaces/:workspaceId/invites/:inviteId`
- `GET /api/workspace-invites`、`POST /api/workspace-invites/resolve`、`POST /api/workspace-invites/:inviteId/accept|decline`、`POST /api/workspace-invites/accept|decline`
- `PATCH|DELETE /api/workspaces/:workspaceId/members/:memberId`、`POST /api/workspaces/:workspaceId/leave`、`POST /api/workspaces/:workspaceId/ownership-transfer`
- `GET /api/workspaces/:workspaceId/deletion-summary`、`DELETE /api/workspaces/:workspaceId`、`GET /api/workspaces/trash`、`POST /api/workspaces/:workspaceId/restore`

### 16.4 验收标准

第一批已通过：

- 用户可在至少两个数据库或本地工作区之间切换，文档内容和活动文档不串联。
- owner 可重命名工作区，当前选择与内容在刷新后恢复。
- 未授权工作区、跨工作区文档、文件 key 和协作 room 均被服务端拒绝。
- IndexedDB v1 数据无损迁移到 v2 `Nexus 工作区`。

M6.2 实现与覆盖：

- 邀请覆盖新用户认证返回、已注册用户接受、重发/撤销/过期和投递失败警告；原始令牌只在邮件 URL 的 fragment 中短暂出现。
- 成员覆盖角色调整、移除、退出、所有权转让和最后一名 owner 保护；真实 PostgreSQL 场景用于验证并发事务。
- 删除覆盖摘要、确认名、回收站、恢复、对象前缀隔离以及对象删除失败后的重试保留。
- 端到端规格覆盖邀请、成员和工作区删除/恢复路径。最终 Docker、真实 PostgreSQL 与完整 Playwright 回归尚未在当前环境重新通过，不能标为完整发布验收。

## 17. M7：真实分享与页面权限（P0）

### 17.1 目标

把当前仅存在于界面的分享选项升级为由服务端强制执行的真实访问能力。

### 17.2 功能范围

- 增加可直接打开的文档路由 `/documents/:documentId`。
- 文档默认继承工作区权限，所有者可设置私有、团队可查看或链接只读。
- 私有文档仅作者、显式授权成员和工作区所有者可访问。
- 链接分享使用随机令牌路由 `/share/:token`，默认只读，可随时关闭或重新生成。
- 支持可选过期时间；分享密码、下载限制和域名白名单暂缓。
- 文件下载、历史版本和协同 WebSocket 使用与文档一致的权限判定。
- 分享创建、关闭、重新生成和访问失败写入脱敏审计。

### 17.5 M7.1 当前状态（2026-07-21）

- 已完成：直接文档路由、私有/团队访问策略、显式 editor/viewer 授权、作者元数据回填、文档级读写与历史恢复、工作区目录过滤、文件/历史/Yjs 权限边界。

- M7.2 尚未开始：匿名分享令牌、只读分享页、过期/撤销、访问审计和签名附件 URL。

## 18. M8：编辑体验、通知与知识发现（P1，暂缓）

### 18.1 结构化富文本

- 将正文从纯文本升级为可版本化的 TipTap/ProseMirror JSON，保留纯文本投影用于搜索。
- 第一批支持粗体、斜体、删除线、行内代码和链接。
- 支持块拖拽排序、多块选择、复制粘贴和批量删除。
- 支持 Markdown 导入导出；导入失败时说明具体行和不支持的结构。
- 为现有纯文本内容提供一次性兼容迁移，迁移前后文本不可丢失。

### 18.2 页面组织与搜索

- 页面树支持父子页面、折叠、移动和面包屑。
- PostgreSQL 全文检索覆盖标题、正文纯文本、任务和评论。
- 搜索结果按当前用户权限过滤，并支持工作区、作者、更新时间和内容类型筛选。
- 支持反向链接和“被哪些页面引用”，关系/公式等复杂数据库能力暂缓。

### 18.3 通知与提及

- 评论支持 @工作区成员，任务负责人变更和 @提及产生站内通知。
- 通知中心支持未读数、全部已读、按类型筛选和跳转到目标块。
- 成员被移除或权限回收后不能继续通过历史通知访问内容。
- 邮件通知只作为可选摘要，第一版不做多渠道推送。

### 18.4 验收标准

- 富文本保存、刷新、历史恢复和多人协同后格式保持一致。
- 500 个普通文本块下滚动、输入和搜索仍可用，不出现明显布局跳动。
- 搜索和通知不会返回无权限工作区或文档的信息。
- @提及可以从通知中心准确定位到文档和块。

## 20. M10：权限感知 AI 工作台（P2，暂缓）

### 20.1 前置条件

- M7 页面权限已由服务端统一执行。
- M8 正文结构和全文检索稳定。
- M9 已具备后台任务、配额、审计和可观测性。

### 20.2 第一批能力

- 选中文本执行改写、缩写、扩写、总结和翻译。
- 将会议记录或文档内容提取为待办，并由用户确认后写入。
- 在当前工作区内进行语义搜索和问答，答案必须附文档与块引用。
- 支持流式输出、取消请求、失败重试和明确的模型/配额错误。

### 20.3 权限与安全

- 检索候选、向量索引、缓存和引用都按当前用户权限过滤。
- 权限变化后及时使相关检索缓存失效。
- 默认不使用用户内容训练模型，不在日志中保存完整提示词和文档正文。
- 模型供应商通过服务端适配层接入，密钥只保存在服务端环境变量或密钥管理服务。
- 第一阶段不允许 AI 自动邀请成员、修改权限、删除页面或执行不可逆操作。

### 20.4 验收标准

- AI 问答只引用当前用户可访问的内容，权限回收后无法继续查询旧引用。
- 每个知识回答至少包含一个可打开的文档/块引用；无依据时明确说明未找到。
- AI 生成的任务在用户确认前不写入正式文档。
- 超时、限流、供应商不可用和配额耗尽都有明确提示，且不会丢失原文。
