## 2. 项目背景

本项目是一个面向学习和作品展示的现代协作编辑器练手项目，核心目标是理解并实现类似 Notion 的块编辑体验，以及多人实时协同编辑背后的数据一致性机制。

项目涉及两个主要技术难点：

- 灵活的数据模型：一切内容抽象为 Block，支持段落、标题、待办、图片等不同类型，并为后续嵌套块、拖拽排序、页面树扩展留下空间。
- 分布式数据一致性：多人同时编辑同一页面时，通过 CRDT 技术同步内容，减少手写冲突解决逻辑。

## 3. 产品目标

### 3.1 长期目标

构建一个支持块编辑、实时协同、持久化存储和基础权限控制的协作编辑应用。

### 3.2 阶段目标

第一阶段优先交付一个可本地运行、可实际编辑的单机版块编辑器，验证编辑体验和块数据模型；之后再逐步引入协同后端、数据库持久化和进阶功能。

### 3.3 成功标准

- 用户可以创建和编辑由多个块组成的页面。
- 用户可以新增、删除、修改、排序基础块。
- 页面内容可以本地保存并在刷新后恢复。
- 后续阶段中，两个浏览器窗口可以实时同步同一页面内容。
- 数据模型清晰，能够扩展到嵌套块、协同编辑和数据库持久化。

## 4. 用户与使用场景

### 4.1 目标用户

- 希望学习富文本编辑器、CRDT、实时协同和树形数据建模的开发者。
- 希望通过一个完整项目展示全栈能力的开发者。

### 4.2 核心场景

- 用户打开编辑器，创建或继续编辑一个文档页面。
- 用户通过块结构组织内容，例如标题、段落和待办事项。
- 用户调整块顺序，快速组织页面结构。
- 在协同阶段，多个用户打开同一页面并实时看到彼此编辑结果。

## 5. 产品范围

### 5.1 第一阶段 MVP 范围

第一阶段聚焦单机版块编辑器，不接入后端协同服务。

必须包含：

- 基础编辑页面。
- 三种基础块：段落、标题、待办。
- 块的新增、删除、编辑。
- 块的排序，优先支持简单上移/下移；如果实现成本可控，再支持拖拽排序。
- 本地持久化，优先使用 IndexedDB。
- 页面刷新后恢复编辑内容。

暂不包含：

- 多人实时协同。
- 登录、权限和用户体系。
- PostgreSQL、Redis、对象存储。
- 历史版本。
- 图片、文件、表格、看板等复杂块。

> 说明：以上是第一阶段 MVP 的历史排除范围。当前项目已推进到 M5，这些能力均已有实现；详见第 14 节当前实现状态。

### 5.2 第二阶段范围：协同雏形

- 引入 Yjs 作为 CRDT 引擎。
- 引入 WebSocket 协同服务。
- 两个浏览器窗口打开同一页面时可以同步文本内容。
- 实现基础在线状态或光标展示。
- 暂不处理复杂权限和长期历史版本。

### 5.3 第三阶段范围：后端与持久化

- 引入 Node.js 后端服务。
- 将页面和块数据保存到 PostgreSQL。
- 设计块的树形关系存储结构。
- 支持页面初次加载时从数据库恢复数据。
- 增加简单用户和页面归属关系。

### 5.4 第四阶段范围：进阶能力

- 嵌套块。
- 图片或文件块。
- Slash Command。
- 历史版本。
- 离线编辑与重连同步。
- 大页面性能优化，例如虚拟滚动。

## 6. 功能需求

### 6.1 页面编辑

- 用户进入应用后默认看到一个编辑页面。
- 页面由多个块组成。
- 页面内容为空时显示可编辑的起始块。
- 用户输入内容后，块内容实时更新到应用状态。

### 6.2 块类型

第一阶段支持三种块类型：

- 段落块：用于普通文本。
- 标题块：用于一级或二级标题，第一阶段可先支持单一标题样式。
- 待办块：包含勾选状态和文本内容。

后续可扩展：

- 图片块。
- 文件块。
- 表格块。
- 看板块。
- 嵌套容器块。

### 6.3 块操作

第一阶段支持：

- 在当前块后新增块。
- 删除当前块。
- 修改块类型。
- 编辑块内容。
- 调整块顺序。

建议交互：

- 回车创建新段落块。
- 空块退格删除当前块或合并到前一个块。
- 待办块支持勾选和取消勾选。

### 6.4 本地持久化

- 编辑内容需要保存到浏览器本地。
- 刷新页面后恢复上次编辑内容。
- 本地存储结构需要与后续服务端数据模型尽量保持一致。

### 6.5 实时协同

第二阶段支持：

- 多个客户端加入同一页面房间。
- 一个客户端编辑文本后，其他客户端实时收到更新。
- 使用 Yjs 自动处理并发编辑冲突。
- 通过 awareness 机制同步光标或用户在线状态。

### 6.6 后端 API

第三阶段可提供 REST API：

- `POST /api/pages`：创建页面。
- `GET /api/pages/:pageId/blocks`：获取页面下所有块。
- `POST /api/blocks/:blockId/move`：移动块到新位置。
- `GET /api/history/:pageId`：获取页面历史版本。

第三阶段可提供 WebSocket API：

- 连接地址：`ws://host/collab/page_:pageId`
- 消息类型：`yjs-update`
- 消息类型：`awareness`

## 7. 数据模型

### 7.1 Block 基础结构

```json
{
  "id": "block_uuid_1",
  "type": "paragraph",
  "parent": "page_uuid_123",
  "children": ["block_uuid_2", "block_uuid_3"],
  "content": {
    "text": "这是一个段落",
    "richText": [
      { "text": "这是一个", "bold": true },
      { "text": "段落" }
    ]
  },
  "createdBy": "user_1",
  "updatedAt": 1678886400000,
  "version": 15
}
```

### 7.2 第一阶段本地模型

第一阶段可以简化为：

```ts
type BlockType = "paragraph" | "heading" | "todo";

interface Block {
  id: string;
  type: BlockType;
  content: string;
  checked?: boolean;
  parentId: string | null;
  children: string[];
  createdAt: number;
  updatedAt: number;
}
```

### 7.3 后续数据库模型

后续服务端建议采用闭包表加物化路径组合：

- `blocks`：存储块的基础属性和内容。
- `block_relationships`：存储祖先、后代和层级深度。
- `pages`：存储页面元数据。
- `users`：存储用户基础信息。

## 8. 技术方案

### 8.1 推荐技术栈

第一阶段：

- 前端框架：React
- 编辑器内核：TipTap
- 本地存储：IndexedDB
- 构建工具：Vite
- 测试：Vitest + Testing Library

第二阶段：

- 协同引擎：Yjs
- 实时通信：WebSocket 或 y-websocket
- 协同服务：Node.js

第三阶段：

- 后端：Node.js
- 数据库：PostgreSQL
- 可选缓存：Redis
- 可选协同框架：Hocuspocus

### 8.2 技术选型理由

- TipTap 基于 ProseMirror，生态成熟，适合快速实现富文本编辑能力。
- Yjs 是成熟 CRDT 库，可以降低多人协同冲突处理复杂度。
- IndexedDB 适合第一阶段本地持久化，避免过早引入后端复杂度。
- PostgreSQL 适合后续存储页面、块和树形关系。

## 9. 非功能需求

### 9.1 性能

- 第一阶段应支持至少 200 个块的流畅编辑。
- 后续阶段需要关注大量块渲染性能，必要时引入 React.memo 或虚拟滚动。

### 9.2 可扩展性

- 块类型需要通过清晰的数据结构扩展。
- 编辑器状态和持久化逻辑需要隔离，便于从 IndexedDB 迁移到服务端。
- 协同逻辑不应污染基础块编辑逻辑。

### 9.3 可靠性

- 本地保存失败时需要保留内存中的编辑内容。
- 协同阶段网络断开时，应提示离线状态，并在恢复后重新同步。

### 9.4 可测试性

- 块 reducer 或状态操作函数需要覆盖单元测试。
- 本地存储适配层需要覆盖保存和恢复测试。
- 编辑器关键交互需要覆盖组件测试。

## 12. 风险与约束

- 富文本编辑器交互复杂，不建议从零实现 contenteditable。
- CRDT 算法复杂，不建议自研协同算法。
- 第一阶段若过早引入后端、权限和数据库，会拖慢 MVP 验证。
- Yjs 二进制文档与结构化数据库之间存在同步复杂度，第三阶段需要专门设计。
- 嵌套块和拖拽排序会显著增加状态管理复杂度，应在基础编辑稳定后再做。
